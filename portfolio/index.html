Decreased churn rate with a new tracking mode

🛒 Product

TrackiPet

🏰 Domain

IoT (wearable trackers)

👩 Users

Dog owners

🗺 Market

Europe (Amazon Europe), B2C

📱 Platforms

iOS and Android

👨‍💻 My role

An in-house senior product designer

⏰ Project duration

September 2022 – December 2022

🗂 Project type

New functionality

⛳️ Business goal

Decrease churn rate of new users

🛠 Design process

Prepared task in Jira by a template. Added a project in Figma by a template. Linked everything together

A template, which reflects my work processes

Had a discussion with a product manager and documented a context

  • At the time, there was an option in apps to track only in minutes (1 minute, 5 minutes, 10 minutes, 15 minutes, 30 minutes, 45 minutes, 60 minutes, 120 minutes, 240 minutes). It was enough for use cases in Trackimo+ apps
  • According to several internal and external user interviews made by product managers, in such specific task (track pets), time plays a significant role in finding a pet as a moving object. And even 1 minute delay is too much for this task
  • To decrease churn, presumably, we need to provide users with an essential functionality: tracking in seconds
  • New tracking mode could be also used as a core to define new subscription plans and business model update
😥 User needs formulated as a user story

As a user, I want to see precise and up-to-date information regarding my pet location, so that I’ll be sure that my pet is in a safe zone

Had a discussion with development team leads and documented an additional context

  • New mode requires exceptional indication as it should be limited by 15 minutes because of hardware limitations (battery drainage) and software limitation (high server load and package traffic limit)
🔩 Technical limitations
  • 15 minutes per session
  • 5 activations per day

Defined a design solution

Provide users with:

  • An option to activate "Live" tracking mode
  • An indication of time left for an activation
  • An indication of number of feature activations left for the current day
  • An information on when number of feature activations resets

Analyzed how switching of tracking modes works now in iOS and Android apps

A user flow, which reflects existing app functionality

Started with an early concept: Decided where the switcher and indicator of Live tracking mode should be located

🛠 Design process specifics

The product already had a working design library. That’s why wireframes step was omitted

A screen displays position of the new element

🤔 Other options considered
  • In device settings: Declined because of functionality should be easily accessible for daily use
  • In devices panel: Declined because it will lead to bloat
  • At other corners of the visible map: Declined because the element will be too close to other active elements (e.g. "Add a device" button or a modal window)

Decided what icons should be used

Icon options for the element

Defined states (a behavior) for the switch/indicator

Activated functionality (more than 30 seconds left), activated functionality (less than 30 seconds left), default state (1 activation left), default state (none activations left)

Created a first iteration of user flow

Including several corner cases

Asked for feedback on early concept from product manager, developers and QA engineers

⚖️ Tradeoffs

According to internal feedback, the element is too bulky and takes a lot of screen space. To address this issue, after considering several options, we agreed to make a more compact version and add a tutorial.

  • Applied white-label styles and components
  • Defined final name for the functionality ("Live tracking" instead of "Realtime")
  • Simplified the element, made it more compact

    • Icon was removed, new name applied, counter was remade

  • Changed states of the element to combine statuses

    • Default state (5 activations left), default state (2 activations left), none activations left, activated functionality

  • Added, tested and removed notifications when timer starts, ends or stops
  • Added multiple devices behavior

    • Tracking mode affects all turned on devices, so badges of them all are changed to "Live tracking". “Update location of all trackers” button now has “Disabled” state during “Live location”

Asked for feedback on refined concept from active users

  • Added subflow when there are no activations per day left

    • A modal window with a detailed explanation why functionality is inactive at the moment

  • Added a functionality to stop the timer and a confirmation when users stop the timer

    • A modal window with a confirmation to prevent accidental stop of the timer

Refined user flows to include all changes

Nth iteration of a user flow with corner cases

And then:

  • Got developers feedback on refinemens
  • Created ready for development assets
  • Added acceptance criteria
  • Together with a product manager defined events for analytics
  • Supported developers during the implementation
  • Updated design library
📊 Results
  • Most of monthly active users from beta testing group tried “Live tracking” functionality
  • Most of them were using “Live tracking” functionality at least once per two days (2-month statistics)
  • Churn rate decreased by 7% for the period of observation
📆 Next steps planned
  • Track feature usage events to get more insights
  • Include functionality into subscription flows update (focused on ARPU)

Improved speed and quality of knowledge transfer from experts to engineers

🛒 Product

An internal project of Gira company

🏰 Domain

IoT (smart home technologies)

👩 Users

Engineers and experts (customer support specialists)

🗺 Market

Europe (Germany), B2C/B2B

📱 Platforms

iOS and web

👨‍💻 My role

A contract product designer as a part of one of cross-functional teams focused on improving engineers’ and end users’ satisfaction

⏰ Project duration

June – August 2020

🗂 Project type

New product (as MVP)

⛳️ Business goal

Decrease average time engineers spend on-site

🛠 Design process

Had a discussion with a stakeholders and documented a context

  • Gira is a worldwide electrical wire, light and installation company with headquarters in Germany. It started to digitally transform their business around 10 years ago
  • While solving wider range of customer problems with smart home technologies, business goals awared of on-site engineers problems and focused on increasing quality of engineers' work, their work satisfaction, and building community of happy customers, engineers and experts to increase customer lifetime and retention
  • COVID-19 pandemic forced this transformation, as additional in-person communication were preferably shortened or avoided to stop spreading the virus
📊 Metrics: measures of success¹
  • Average time an engineer spends on-site
  • Engineer satisfaction of expert service
  • End customer satisfaction of engineer service
  • End customer lifetime

¹ Definitions of success were not set

Defined groups of people involved

  • Stakeholders: managers of the company
  • Experts: the company's customer support specialists, providing service for engineers and end customers
  • Engineers: electric engineers and architects in the company, in other service companies or freelance engineers, providing service for end customers
  • End customers: house or apartment owners, office managers, store managers

Prepared questions and conducted remote interviews with engineers and experts

Totally there were 5 external interviews conducted with engineers and 5 external interviews with experts. 2 internal interviews with stakeholders were conducted earlier

Analyzed interview outcomes and defines problems engineers do have regarding the business goal

  • Using voice only (via phone) is not sufficient for fast support
  • Waiting time on hotline is unpredictable for engineers
  • Setting up TeamViewer Pilot to communicate with experts takes time and a lot of effort

Defined hypothesis

  • If we will solve a problem of low quality and low speed of knowledge transfer from experts to engineers
  • with a fast, simple, exclusive product for remote video assistance,
  • we will increase reduce average time engineer spends on-site¹
  • according to problems analysis from conducted interviews

¹ Also we believed this release will affect engineer satisfaction of expert service, end customer satisfaction of engineer service and end customer lifetime

Defined user goals

First iteration displaying start and end points, and user goals as checkpoints

Defined happy path

Second iteration with a happy path: an ideal route for users to get value

Defined screen elements and corner cases

Third iteration with additional paths including errors, and information architecture including screens and screen elements

Defined wireframes

Fourth iteration with wireframes

Also, a low-fidelity interactive prototype was created

Conducted usability testing

  • There are problems perceiving English interface by engineers and experts having German language as native — we translated the interface into German
  • Unclear when a progress of calling an expert continued or freezed (e.g. due to some internal error) — we added animated bar indicating the progress
  • Unclear how long will it take to wait for a start of a call with an expert — we agreed on adding a number in a queue and planned to display estimated time left until the call start, when there will be enough data collected to calculate means according to a time of a day
  • For both engineers and experts it was hard to point to specific object like switches, wires or toggles (and sometimes it's just dangerous to do with powered lines) — planned to add an ability to interactively draw on video screen/photo to underline objects
  • It was hard to understand what rate scale from 1 to 5 means (how to mark the best and the worst experience) — switched text buttons to understandable icons

Nth iteration of user flows with screens designed in high resolution

  • We defined scenarios and acceptance criteria. Having extensive flowchart at the moment helped us to describe them faster and clearer
  • I prepared production-ready files in Figma with documented styles and patterns
  • During the development handover and support, additional corner cases were discovered, so user flows and documentation were updated accordingly
