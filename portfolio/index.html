Prepared task in Jira by a template. Added a project in Figma by a template. Linked everything together

A template, which reflects my work processes

Had a discussion with a product manager and documented a context

  • At the time, there was an option in apps to track only in minutes (1 minute, 5 minutes, 10 minutes, 15 minutes, 30 minutes, 45 minutes, 60 minutes, 120 minutes, 240 minutes). It was enough for use cases in Trackimo+ apps
  • According to several internal and external user interviews made by product managers, in such specific task (track pets), time plays a significant role in finding a pet as a moving object. And even 1 minute delay is too much for this task
  • To decrease churn, presumably, we need to provide users with an essential functionality: tracking in seconds
  • New tracking mode could be also used as a core to define new subscription plans and business model update
üò• User needs formulated as a user story

As a user, I want to see precise and up-to-date information regarding my pet location, so that I‚Äôll be sure that my pet is in a safe zone

Had a discussion with development team leads and documented an additional context

  • New mode requires exceptional indication as it should be limited by 15 minutes because of hardware limitations (battery drainage) and software limitation (high server load and package traffic limit)
üî© Technical limitations
  • 15 minutes per session
  • 5 activations per day

Defined a design solution

Provide users with:

  • An option to activate "Live" tracking mode
  • An indication of time left for an activation
  • An indication of number of feature activations left for the current day
  • An information on when number of feature activations resets

Analyzed how switching of tracking modes works now in iOS and Android apps

Started with an early concept: Decided where the switcher and indicator of Live tracking mode should be located

ü§î Other options considered
  • In device settings: Declined because of functionality should be easily accessible for daily use
  • In devices panel: Declined because it will lead to bloat
  • At other corners of the visible map: Declined because the element will be too close to other active elements (e.g. "Add a device" button or a modal window)

Decided what icons should be used

Defined states (a behavior) for the switch/indicator

Created a first iteration of user flow. Including several corner cases

  • 5, 4, 3, 2, 1 activations left
  • None activations left
  • Timer starts
  • Less than 30 seconds of timer left
  • Timer ends

Asked for feedback on early concept from product manager, developers and QA, considered and implemented through feedback loops

According to internal feedback, the element is too bulky and takes a lot of screen space. To address this issue, after considering several options, we agreed to make a more compact version and add a tutorial.

  • Applied white-label styles and components
  • Defined final name for the functionality ("Live tracking" instead of "Realtime")
  • Simplified the element, made it more compact
  • Changed states of the element to combine statuses
  • Added, tested and removed notifications when timer starts, ends or stops

Asked for feedback on refined concept from active users

  • Added subflow when there are no activations per day left
  • Added a functionality to stop the timer and a confirmation when users stop the timer
